你是一名资深 Go 后端工程师和 MQTT 协议专家。请仿照 EMQX 官方性能测试工具 emqtt-bench，使用 Go 语言开发一个跨平台 MQTT 压测工具。

项目目标：
开发一个 Go 版本的 MQTT Benchmark CLI 工具，名称暂定为 go-mqtt-bench。它应尽量复刻 emqtt-bench 的核心使用模型，但不要求完全逐项兼容 Erlang 实现。优先支持 Windows、Linux、macOS，重点解决官方 emqtt-bench 没有 Windows 原生构建包的问题。

参考对象：
1. EMQX 官方文档中的 eMQTT-Bench 使用方式。
2. GitHub 项目 emqx/emqtt-bench 的 README 参数模型。
3. 官方工具的三个核心子命令：
    - conn：创建大量 MQTT 连接，用于连接量压测。
    - sub：创建大量订阅客户端，订阅主题并统计接收速率。
    - pub：创建大量发布客户端，按指定速率发布消息并统计发送速率。

必须实现的 CLI 子命令：
1. go-mqtt-bench conn
2. go-mqtt-bench sub
3. go-mqtt-bench pub

基础技术要求：
1. 使用 Go 1.22+。
2. MQTT 客户端库优先选择 Eclipse Paho MQTT Go Client；如果你认为其他库更适合高并发压测，需要说明理由。
3. 使用 cobra 或 urfave/cli 实现命令行。
4. 使用 zap、zerolog 或标准 log/slog 实现日志。
5. 使用 context 控制生命周期。
6. 使用 goroutine + worker/rate limiter 实现并发客户端创建和消息发布。
7. 所有统计指标必须使用 atomic 或无锁/低锁方案，避免统计逻辑成为性能瓶颈。
8. 工具必须能在 Windows、Linux、macOS 编译运行。
9. 代码结构要清晰，便于后续扩展 WebSocket、TLS、Prometheus、配置文件等能力。

核心命令设计：

一、conn 命令：连接压测

命令示例：
go-mqtt-bench conn -h 127.0.0.1 -p 1883 -c 10000 -R 1000
go-mqtt-bench conn -h emqx-server -c 1000
go-mqtt-bench conn -h 192.168.0.99 -c 50000 --ifaddr 192.168.0.100

功能要求：
1. 创建指定数量的 MQTT 客户端连接。
2. 支持控制连接建立速率。
3. 支持连接成功、连接失败、重连、断开统计。
4. 支持连接建立完成后保持连接，不立即退出。
5. 支持 Ctrl+C 优雅退出，退出时主动断开所有客户端。
6. 支持连接失败重试。

参数要求：
通用参数：
- -h, --host：MQTT Broker 地址，默认 localhost。允许逗号分隔多个 host。
- -p, --port：MQTT Broker 端口，默认 1883。
- -V, --version：MQTT 协议版本，支持 3、4、5；默认 5。其中 3 表示 MQTT 3.1，4 表示 MQTT 3.1.1，5 表示 MQTT 5.0。
- -c, --count：客户端数量，默认 200。
- -R, --connrate：每秒创建连接数，默认 0。若为 0，则使用 --interval。
- -i, --interval：创建客户端的时间间隔，单位毫秒，默认 10。
- --ifaddr：本地绑定 IP 地址或网卡地址。需要支持单个 IP；后续可扩展为逗号分隔多个 IP。
- --prefix：Client ID 前缀。
- --shortids：使用短 Client ID。
- -n, --startnumber：客户端序号起始值，默认 0。用于多进程/多压力机场景避免 Client ID 冲突。
- --num-retry-connect：连接建立失败后的重试次数，默认 0。
- --reconnect：断线后的最大重连次数，0 表示禁用，默认 0。
- -u, --username：用户名。
- -P, --password：密码。
- -k, --keepalive：Keep Alive 秒数，默认 300。
- -C, --clean：Clean Session / Clean Start，默认 true。
- -x, --session-expiry：MQTT 5 Session Expiry 秒数，默认 0。
- -S, --ssl：启用 TLS，默认 false。
- --cacertfile：CA 证书路径。
- --certfile：客户端证书路径。
- --keyfile：客户端私钥路径。
- --ws：启用 WebSocket 连接，MVP 阶段可先保留参数但明确返回“not implemented”。
- --quic：启用 QUIC，MVP 阶段不实现，只保留参数并提示暂不支持。
- --prometheus：开启 Prometheus 指标。
- --restapi：开启 REST API 或 /metrics 服务，例如 ":9090" 或 "127.0.0.1:9090"。
- --log-to：日志输出位置，支持 console/null，默认 console。

Client ID 生成规则：
1. 如果提供 --prefix：
    - 默认生成：<prefix><sequence>
    - 如果需要避免重复，可允许配置是否追加随机数。
2. 如果未提供 --prefix：
    - 默认生成：<hostname>_bench_<cmd>_<random>_<sequence>
3. 如果启用 --shortids：
    - 若提供 prefix，则生成：<prefix><sequence>
    - 若未提供 prefix，则直接使用 sequence。
4. sequence 从 --startnumber 开始递增。

二、sub 命令：订阅压测

命令示例：
go-mqtt-bench sub -t t -h emqx-server -c 500
go-mqtt-bench sub -c 50000 -i 10 -t bench/%i -q 2
go-mqtt-bench sub -t bench/%i -h 127.0.0.1 -c 1000 -q 0

功能要求：
1. 创建指定数量的 MQTT 客户端。
2. 每个客户端连接成功后订阅指定 topic。
3. 支持 QoS 0、1、2。
4. 接收消息时统计：
    - total received messages
    - current receive rate，单位 msg/sec
    - bytes/sec
    - per-client 或全局接收计数
5. 每隔固定时间，例如 1 秒，输出类似：
   recv(total): total=2102563, rate=99725(msg/sec), bytes_rate=xx(MB/sec)
6. 支持 topic 占位符：
    - %c：Client ID
    - %u：Username
    - %i：客户端序号
7. 支持 payload header 解析：
    - cnt64：检查每个发布者消息计数是否严格递增。
    - ts：解析发布时间戳，用于计算端到端延迟。
      MVP 阶段可以先实现 ts 延迟统计，cnt64 可作为二期能力。

sub 额外参数：
- -t, --topic：订阅主题，必填。支持 %u、%c、%i 变量。
- -q, --qos：订阅 QoS，默认 0。
- --payload-hdrs：接收端解析 payload header，支持 cnt64、ts，逗号分隔。

三、pub 命令：发布压测

命令示例：
go-mqtt-bench pub -t t -h emqx-server -s 16 -q 0 -c 10 -I 10
go-mqtt-bench pub -c 100 -I 10 -t bench/%i -s 256
go-mqtt-bench pub -t t -h 192.168.0.99 -c 20 -I 100

功能要求：
1. 创建指定数量的 MQTT 发布客户端。
2. 每个客户端按指定间隔持续发布消息。
3. 支持控制单客户端发布间隔。
4. 支持限制总发布数量。
5. 支持 QoS 0、1、2。
6. 支持 Retain 标志。
7. 支持 payload size。
8. 支持固定消息内容。
9. 支持 topic 占位符。
10. 支持发布统计：
    - total published messages
    - publish rate，单位 msg/sec
    - bytes/sec
    - success count
    - failure count
    - inflight count，QoS 1/2 场景
11. 每秒输出类似：
    send(total): total=100000, rate=10000(msg/sec), bytes_rate=2.44(MB/sec), success=..., failed=...
12. 支持等待所有 publisher 至少完成连接尝试后再开始发布。

pub 额外参数：
- -I, --interval-of-msg：单客户端发布消息间隔，单位毫秒，默认 1000。
- -t, --topic：发布主题，必填。支持 %u、%c、%i、%s 变量。
- -s, --size：Payload 大小，默认 256 字节。
- -m, --message：指定消息内容。可以是字面量，也可以扩展支持 template://<file_path>。
- --topics-payload：JSON 文件，定义多个 topic 和 payload，MVP 阶段可先不实现，但需要预留接口。
- -q, --qos：发布 QoS，默认 0。
- -r, --retain：Retain 标志，默认 false。
- -L, --limit：最大发布消息数，0 表示无限制，默认 0。
- -F, --inflight：QoS 1/2 最大飞行窗口，0 表示无限制，默认 1。
- -w, --wait-before-publishing：等待所有发布客户端至少尝试连接后再开始发布，默认 false。
- --max-random-wait：每个 publisher 开始发布前的最大随机等待毫秒数。
- --min-random-wait：每个 publisher 开始发布前的最小随机等待毫秒数。
- --retry-interval：QoS 1/2 未收到确认时的重发间隔，MVP 阶段可先不实现，保留参数。

topic 占位符规则：
实现函数 RenderTopic(template string, client ClientInfo) string：
- %c 替换为 Client ID。
- %u 替换为 Username。
- %i 替换为客户端序号。
- %s 替换为字符串形式的序号或可配置 shard/sequence。
  要求：
1. 占位符替换必须高效，避免每条消息都重复高成本分配。
2. 对 pub 场景，topic 通常在客户端启动时即可渲染并缓存。
3. 如果未来支持每条消息动态 topic，再单独扩展。

Payload 设计：
1. 默认 payload 为指定 size 的字节数组。
2. 如果指定 --message，则使用指定消息内容。
3. 如果指定 --payload-hdrs：
    - cnt64：在 payload 前部写入 uint64 自增计数。
    - ts：在 payload 前部写入发送时间戳，建议使用 UnixNano int64。
4. 如果 payload size 小于 header 长度，应返回参数错误。
5. 需要保证 header 格式在 pub/sub 两端一致。

6. disconnect_total。
7. pub_total。

9. pub_failed_total。
10. pub_rate。
11. pub_bytes_total。
12. pub_bytes_rate。
13. sub_recv_total。

16. sub_recv_bytes_rate。
17. latency_min。

20. latency_p50、p90、p95、p99，可使用 hdrhistogram-go 或自定义近似统计。

22. out_of_order_total，cnt64 二期实现。

统计输出要求：
1. 默认每 1 秒输出一次。
2. 日志格式简洁，适合终端观察。
3. 支持 --log-to null 静默模式。

核心抽象：
1. Config：保存通用参数。
2. ConnConfig、PubConfig、SubConfig：分别保存子命令参数。
3. ClientInfo：
    - Index int64
    - ClientID string
    - Username string
    - Host string
    - LocalAddr string
4. Runner 接口：
   type Runner interface {
   Run(ctx context.Context) error
   }
5. RateLimiter：
    - 支持 connrate，每秒 N 个连接。
    - 支持 interval，每隔 N 毫秒创建一个连接。
    - connrate 优先级高于 interval。
6. StatsCollector：
    - 使用 atomic.Int64 / atomic.Uint64。
    - 避免每条消息都加锁。
7. Reporter：
    - 每秒快照当前统计。
    - 基于上次快照计算 rate。

连接模型：
1. 每个 MQTT 客户端对应一个 goroutine。
2. 主 Runner 按 rate limiter 创建客户端。
3. 客户端创建后执行 connect。
4. conn 命令：连接成功后阻塞等待 ctx.Done。
5. sub 命令：连接成功后订阅 topic，消息回调中更新统计。
6. pub 命令：连接成功后进入发布循环。
7. 退出时：
    - 停止创建新客户端。
    - 停止发布。
    - 取消订阅可选。
    - Disconnect 所有已连接客户端。
    - 输出最终统计摘要。

连接失败处理：
1. connect 失败时记录失败计数。
2. 如果 --num-retry-connect > 0，则按固定退避或指数退避重试。
3. 如果重试仍失败，客户端退出。
4. 如果 --reconnect > 0，则断线后允许重连。
5. 所有错误都需要带上 client index/client id/host 信息，便于定位。

多 host 处理：
1. --host 支持逗号分隔。
2. 客户端按 index 对 host 做轮询分配。
3. 后续可以扩展权重分配。

本地 IP 绑定：
1. --ifaddr 支持绑定本地 IP。
2. 在 Go 网络层通过 net.Dialer.LocalAddr 实现。
3. Windows 下如果绑定 IP 不存在，需要返回明确错误。
4. 注意单源 IP 的 TCP 临时端口限制。工具应在启动时给出警告：单个源地址不建议超过 64K 连接。
5. 对 Linux 用户提示可调整：
    - ulimit -n
    - net.ipv4.ip_local_port_range
    - net.ipv4.tcp_tw_reuse 等

TLS 支持：
1. -S/--ssl 启用 TLS。
2. --cacertfile 加载 CA 证书。
3. --certfile 和 --keyfile 加载客户端证书。
4. 支持跳过证书校验的参数可以作为二期能力，例如 --insecure-skip-verify，但默认必须安全。
5. --ssl-version 可作为二期能力，MVP 先使用 Go 默认 TLS 配置。

WebSocket / QUIC：
1. --ws 参数保留。
2. --quic 参数保留。
3. MVP 阶段不要求实现 QUIC。
4. 如果用户指定 --quic，应返回清晰错误：QUIC transport is not implemented in this version。
5. WebSocket 可以作为二期能力，基于 MQTT over WebSocket 实现。

性能注意事项：
1. 不要为每条消息创建大量临时对象。
2. payload 尽量复用。
3. topic 尽量预渲染。
4. 日志不能在高频路径输出。
5. 统计必须低开销。
6. 发布循环要避免 time.Tick 泄漏，使用 ticker 并在退出时 Stop。
7. 大量客户端场景下要避免无限 goroutine 堆积。
8. 支持设置 GOMAXPROCS，默认使用 runtime.NumCPU。
9. 对高连接数测试，文档中提示用户需要调大文件描述符、端口范围、网卡/IP 数量。
10. 单源 IP 不建议超过约 64K 连接，因为 TCP 临时端口数量有限。

典型验证场景：
1. 连接量测试：
   go-mqtt-bench conn -h 127.0.0.1 -p 1883 -c 1000 -R 100
   预期：成功建立 1000 个连接，终端显示 active=1000。

2. 订阅测试：
   go-mqtt-bench sub -h 127.0.0.1 -t bench/%i -c 500 -q 0
   预期：500 个客户端分别订阅 bench/0 到 bench/499。

3. 发布测试：
   go-mqtt-bench pub -h 127.0.0.1 -t bench/%i -c 100 -I 10 -s 256 -q 0
   预期：100 个客户端，每个客户端约 100 msg/sec，总发布速率约 10000 msg/sec。

4. 吞吐测试：
   终端 A：
   go-mqtt-bench sub -h 127.0.0.1 -t t -c 500 -q 0

   终端 B：
   go-mqtt-bench pub -h 127.0.0.1 -t t -c 20 -I 100 -s 256 -q 0

   预期：订阅端显示接收速率，发布端显示发送速率。
